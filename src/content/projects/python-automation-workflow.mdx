## Problem / Context

The operations team at a B2B services company ran a weekly process that looked like this:

1. Export data from three different internal systems
2. Manually reconcile records in Excel
3. Identify discrepancies and send correction emails
4. Generate a summary report and upload it to Google Sheets
5. Notify account managers via Slack

This took 3–4 hours every week, was done by one person, and had an error rate that caused downstream issues. It was also a single point of failure — if that person was on holiday, the process stopped.

## Approach

I broke the process into discrete, testable stages and automated each one:

1. **Extract**: Pull data from each source system via SQL queries and CSV exports
2. **Transform**: Pandas pipeline for cleaning, normalizing, and reconciling records
3. **Validate**: Rule-based checks that flag anomalies before any data moves further
4. **Load**: Write results to PostgreSQL, update Google Sheets
5. **Notify**: Post structured Slack summary with any flagged items

Each stage runs as a Celery task, scheduled via Celery Beat. Failures are caught, logged, and trigger an alert — the team always knows what happened and why.

## Architecture / Technical Decisions

```
Celery Beat (scheduler)
    │
    ▼
Task: extract_data()
    │
    ▼
Task: transform_and_validate()
    │
    ├── Validation pass → Task: load_results()
    └── Validation fail → Alert task → Slack notification
```

Key decisions:
- **Celery over cron**: Needed retries, task visibility, and alerting. Cron + shell scripts would have been fragile.
- **Validation before load**: Catching data quality issues early prevents bad data from propagating. The validation step runs ~20 business rules.
- **Idempotent tasks**: Every task can be re-run safely. This made debugging and re-processing trivial.
- **Structured logging**: JSON logs throughout, collected in a simple admin dashboard. The team can see exactly what ran, when, and what happened.

## Challenges

The main challenge was **handling inconsistent source data**. The legacy systems exported data in slightly different formats depending on who ran the export and when. I spent significant time building robust parsers with fallback logic, and added data lineage tracking so issues are always traceable.

Another challenge was **stakeholder trust**. The team was used to manually reviewing everything. I addressed this by making the validation report the most prominent output — it explicitly lists what was checked and what passed, giving them confidence to let the automation run unattended.

## Outcome / Impact

- 15+ hours of manual work per week eliminated
- Error rate reduced from ~8% to under 0.5%
- Process now runs automatically at 6 AM every Monday, reports arrive before the team starts work
- The same framework was extended to automate two additional reporting processes
