## Problem / Context

The client processed weekly batches of 200–500 documents — analyst reports, regulatory filings, client emails — extracting 30+ specific data fields into a structured research database.

This was done entirely by hand. It took 3–4 days per batch, and despite careful work the error rate was still significant enough to require a separate QA pass.

## Approach

I built an extraction pipeline structured in three stages:

1. **Document ingestion**: Parse PDFs, DOCX, and plain text into normalized text chunks
2. **LLM extraction**: Use a structured extraction prompt with Pydantic output schemas to extract specific fields
3. **Validation and confidence scoring**: Each extracted field gets a confidence score; low-confidence fields are flagged for human review

The human review queue means the system is never fully autonomous on the first version — it's a human-in-the-loop design that earns trust incrementally.

## Architecture / Technical Decisions

```
Document upload (API)
    │
    ▼
Ingestion worker (PDF/DOCX parsing, chunking)
    │
    ▼
Extraction worker (LLM + Pydantic schemas)
    │
    ├── High confidence → auto-commit to DB
    └── Low confidence → human review queue
                             │
                             ▼
                         Analyst reviews → commits
```

Key decisions:
- **Pydantic output schemas**: Using `instructor` library with OpenAI function calling to get structured, typed outputs. This eliminated most post-processing and made validation trivial.
- **Field-level confidence scores**: Rather than document-level confidence, I track confidence per field. This means partially confident documents still get significant value.
- **Two-pass extraction**: A fast first pass extracts the easy fields; a second focused pass targets complex, context-dependent fields. Cheaper and more accurate than one complex prompt.
- **Idempotent processing**: Every document gets a hash. Re-uploading the same document is a no-op.

## Challenges

The main challenge was **prompt brittleness across document types**. A prompt tuned for analyst reports fell apart on regulatory filings. I addressed this with document classification (also LLM-based) and separate extraction prompts per document type.

The second challenge was **hallucination on missing fields**. When a document genuinely doesn't contain a field, some models invent plausible values. I added explicit "field absent" logic and tested extensively with documents where fields were known to be missing.

## Outcome / Impact

- Batch processing time: 3–4 days → under 2 hours
- Extraction accuracy: 94.2% on the held-out validation set (vs. ~91% human baseline)
- Human review queue: ~15% of fields flagged, down from 100% manual review
- Client now processes 3x the document volume without adding headcount
