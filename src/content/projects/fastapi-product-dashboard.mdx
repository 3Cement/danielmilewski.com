## Problem / Context

The client had an early-stage SaaS product with a Flask backend that had grown organically over 18 months. It worked at small scale, but under load the synchronous architecture caused bottlenecks, and the codebase had accumulated enough technical debt to slow down new feature development significantly.

They needed a clean, scalable backend that could support their next 12 months of growth, with a team of 3 developers maintaining it.

## Approach

Rather than incrementally fixing the existing Flask app, we agreed to build a clean FastAPI backend in parallel and migrate endpoints one by one. This let the product stay live throughout the migration.

The new backend was designed around:
- Clear separation of concerns (routes → services → repositories)
- Async-first: `asyncpg` driver, `asyncio`-native throughout
- Feature-based module structure (not layer-based)
- Typed models using Pydantic v2

## Architecture / Technical Decisions

```
FastAPI app
├── routers/          Feature-based route modules
├── services/         Business logic layer
├── repositories/     Database access (async SQLAlchemy)
├── models/           SQLAlchemy ORM models
├── schemas/          Pydantic request/response models
├── core/             Config, auth, dependencies
└── workers/          Background tasks (ARQ)
```

Key decisions:
- **JWT + refresh tokens**: Stateless auth with short-lived access tokens (15 min) and Redis-backed refresh token rotation.
- **RBAC**: Role-based permissions modeled at the database level. Each endpoint declares required permissions as a FastAPI dependency.
- **Background jobs**: Used ARQ (async Redis queue) for report generation and data processing tasks. Kept the request/response cycle fast.
- **Alembic migrations**: Every schema change goes through a migration. No manual DB modifications in any environment.
- **Rate limiting**: Redis-backed rate limiting middleware per route group and per user tier.

## Challenges

The biggest challenge was **multi-tenancy data isolation**. The product serves multiple clients whose data must never mix. I implemented tenant isolation at the SQLAlchemy session level using row-level security policies in PostgreSQL, with tests explicitly verifying cross-tenant isolation.

The second challenge was **migrating existing data** from the old schema to the new one without downtime. We used a dual-write strategy during the transition period and validated data consistency before cutting over.

## Outcome / Impact

- Average API response time: 85ms (down from 210ms)
- Zero downtime during migration
- Backend shipped to production with full test coverage on the auth and billing paths
- The team reports significantly faster feature development on the new codebase
