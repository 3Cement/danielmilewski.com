## Problem / Context

The client's support and onboarding team was growing fast, but internal knowledge was scattered across Confluence, Google Drive, Slack, and Notion. New team members took 2–3 weeks to get up to speed. Senior engineers were constantly interrupted with repetitive questions.

The team needed a way to make institutional knowledge instantly accessible — without forcing everyone to maintain yet another documentation system.

## Approach

Instead of building a traditional search tool, I implemented a RAG pipeline that:

1. **Ingests** documents from Confluence, Notion, and Slack (via their APIs)
2. **Chunks and embeds** content using OpenAI embeddings
3. **Stores** vectors in Pinecone with metadata for filtering and source tracking
4. **Retrieves** semantically relevant chunks at query time
5. **Generates** concise, cited answers via GPT-4

The system is exposed as a Slack bot — the team's existing workflow — so there's zero friction to adoption.

## Architecture / Technical Decisions

```
User query (Slack)
    │
    ▼
FastAPI endpoint (async)
    │
    ├── Query embedding (OpenAI)
    ├── Similarity search (Pinecone, top-k=6)
    ├── Context assembly + prompt construction
    └── LLM call (GPT-4) → streamed response
```

Key decisions:
- **Chunking strategy**: 512-token chunks with 64-token overlap, preserving heading context. Tested multiple strategies — this gave the best retrieval precision.
- **Re-ranking**: Added a lightweight cross-encoder re-ranker before the final LLM call. Improved answer quality noticeably.
- **Source citations**: Every response includes source links. This was non-negotiable for trust and adoption.
- **Caching**: Exact-match query cache with Redis. ~30% of queries are cache hits in practice.

## Challenges

The hardest problem was **handling stale data**. Documents change, get deleted, or become outdated. I implemented a delta-sync mechanism with document versioning, and added "last updated" metadata to every chunk so the LLM can flag potentially outdated information.

The second challenge was **prompt engineering for precision**. Early versions would answer questions using loosely related context. I invested significant time in system prompt design and few-shot examples to prevent hallucination and off-topic responses.

## Outcome / Impact

- Support query resolution time dropped from ~12 minutes to ~3 minutes on average
- New team member ramp-up time reduced from 2–3 weeks to under a week
- Zero complaints about hallucinated answers after the citation system was added
- The client expanded the system to cover their engineering onboarding documentation
